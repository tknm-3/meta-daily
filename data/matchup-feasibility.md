# 相性表（デッキ・マッチアップ表）実現可能性の調査メモ

DLM のメタ結果＋大会（トナメル等）の結果から「どのデッキがどれに勝ったか」を集計して
相性表を作れるか、を CI ランナー（外部アクセス可）から実地調査した結果。生データは
[`probe-matchups.json`](probe-matchups.json)、調査スクリプトは
[`scripts/probe_matchups.py`](../scripts/probe_matchups.py)。

## 結論

公開データだけでは **統計的に意味のある「デッキ×デッキの勝率相性表」は作れない。**
対戦の勝敗自体は取得経路があるが、対戦の大半を「どのデッキ対どのデッキか」に変換できない。

## 判明したデータ経路（すべて CI から HTTP 200 で到達確認）

```
DLM top-decks API (/api/v1/top-decks)
  └→ 各デッキの linkedArticle（大会レポート）
       └→ DLM 記事 API  /api/v1/articles?_id=<id>      … JSON で取得可
            └→ 本文に埋め込まれたトナメル URL           例: tonamel.com/competition/VA23z
                 └→ トナメルのブラケット（対戦表）
```

## 主要な発見

- **DLM 記事に対戦データは無い** … 記事 JSON の `match_like_keys` は空、本文（`parsedMarkdown`）は
  散文＋トナメルへのリンクのみ。勝敗はトナメル側にしかない。
- **トナメルのブラケットは HTML に埋め込まれていない** … SSR ブロブ（`__NUXT__`）は 652 バイトで空。
  データはページ読み込み後に JS が API から後読みする SPA 構成。
- **トナメルの API は GraphQL** … `GET https://tonamel.com/graphql` が **405 Method Not Allowed**
  （404 ではない）を返す＝エンドポイントは存在し POST で動く。「誰が誰に勝ったか＋参加者名」を
  機械的に取得できる経路は **存在する**。

## 相性表化を阻む 2 つの壁

1. **デッキが分かるのは上位入賞者だけ**
   トナメルは「プレイヤー名と勝敗」は持つが「デッキ」を持たない。デッキが判明するのは
   DLM が公開する top cut（1st / 2nd / Top 4・8・16）の数名のみ。1 回戦など大多数の対戦は、
   勝敗は取れてもデッキが不明 → 「デッキ A が デッキ B に勝った」に落とせない。

2. **名寄せが必要 ＆ 単淘で標本が薄い**
   DLM の投稿者名とトナメル参加者名を突合する必要があり崩れやすい。さらに大会は
   単淘・BO3 のため、デッキの組が当たる回数が少なく偏る（総当たりではない）。

## 実現可能な代替案

- **上位限定の対戦ログ**（限定版・小標本）: トナメル GraphQL のクエリを特定し、top cut の
  デッキ判明プレイヤー同士が当たった試合だけ「A が B に勝った」を蓄積。サンプルは小さく上位に偏る。
  トナメルの利用規約（スクレイピング可否）の確認が必要。
- **順位ベースの疑似相性／メタ位置表**（近似版）: トナメル不要。既存の DLM 順位データだけで、
  同一大会内の上位／入賞率から相対的な強さを表にする。真の勝率ではない。

> 調査のみで終了。上記いずれの実装も未着手。
